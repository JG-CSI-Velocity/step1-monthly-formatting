# Deck Polish вЂ” Design Spec

**Date:** 2026-04-17
**Repo:** `JG-CSI-Velocity/ars-production-pipeline`
**Branch (primary):** `feature/deck-polish`
**Status:** Draft вЂ” pending user review

---

## 1. Problem

Every client engagement produces a PowerPoint deck (often 80+ slides) generated by `01_Analysis/00-Scripts/output/deck_builder.py` or assembled from the ~320 analysis cells in `01_Analysis/00-Scripts/analytics/`. The deck's style guide вЂ” fonts, palette, headline rules, layout grid вЂ” is codified in `SLIDE_MAPPING.md`, but output decks drift from the guide. Today the gap is closed by hand, slide by slide, before client delivery.

## 2. Goal

Ship a post-hoc polish pass that reads any PPTX, enforces `SLIDE_MAPPING.md` compliance where safe, and **flags** (rather than rewrites) anything judgment-dependent. Ship on a new branch that does not touch production code.

## 3. Scope decomposition (why this spec is narrow)

This is one of three sibling sub-projects. Each gets its own spec and branch.

| Sub-project | Spec | Branch | Role |
|---|---|---|---|
| **`deck-polish`** | **this spec** | `feature/deck-polish` | Post-hoc PPTX polish. Ships first. |
| `style-system` | future spec | `feature/builder-style-shared` | Extract `style/` to a shared module; refactor `deck_builder.py` to consume it. |
| `html-review` | future spec | `feature/html-review` | Themed HTML aggregator вЂ” consolidates ~320 cells into ~8вЂ“12 consultative review pages. Shares the `style/` module. |

The `style/` module introduced in this spec is designed so the other two sub-projects can import it verbatim.

## 4. Non-goals (this spec)

- Rewriting `deck_builder.py`
- Re-rendering charts (requires source data вЂ” out of polish's reach)
- Auto-rewriting business takeaways (headlines, annotations)
- Building HTML outputs
- Migrating the `style/` module to a shared location (done in `style-system`)

### 5.2 Module boundaries

- `style/` is a pure library вЂ” no file I/O, no pptx mutation side effects. Functions either return values (colors, sizes, scores) or accept a pptx object and mutate it in place via documented APIs.
- `polish.py` is the only code that touches disk or walks a Presentation.
- Everything in `style/` is importable by `deck_builder.py` (via `style-system` branch later) and by the HTML renderer (via `html-review` spec later) without modification.

## 6. Style module

### 6.1 `style/palette.py`

Named `pptx.dml.color.RGBColor` constants from `SLIDE_MAPPING.md`:

```python
NAVY   = RGBColor(0x1B, 0x36, 0x5D)
TEAL   = RGBColor(0x0D, 0x94, 0x88)
CORAL  = RGBColor(0xE7, 0x4C, 0x3C)
AMBER  = RGBColor(0xF3, 0x9C, 0x12)
GRAY   = RGBColor(0x95, 0xA5, 0xA6)
SLATE  = RGBColor(0x6B, 0x72, 0x80)
WHITE  = RGBColor(0xFF, 0xFF, 0xFF)

MUTED_ALPHA = 0.3
```

Helpers:
- `is_palette_color(rgb) -> bool`
- `nearest_palette(rgb) -> RGBColor` (returns closest palette color; used for snap-to-palette)
- `focal(name) -> dict` вЂ” returns `{"focal": RGBColor, "muted": list[RGBColor]}` for charts that need one series emphasized

### 6.2 `style/typography.py`

One factory per text-role from `SLIDE_MAPPING.md`:

```python
slide_title() -> TextStyle("Montserrat", 20, "ExtraBold")
subtitle()    -> TextStyle("Montserrat", 14, "Regular")
chart_title() -> TextStyle("Montserrat", 18, "Bold")
chart_subtitle() -> TextStyle("Montserrat", 12, "Regular")
data_label()  -> TextStyle("Montserrat", 11, "Regular")
axis_label()  -> TextStyle("Montserrat", 10, "Regular")
footnote()    -> TextStyle("Montserrat", 9,  "Regular")
kpi_hero()    -> TextStyle("Montserrat", 36, "ExtraBold")
kpi_label()   -> TextStyle("Montserrat", 11, "Regular")
```

`TextStyle` is a dataclass carrying `font_name`, `size_pt`, `weight`.

Helper: `apply(text_frame, style: TextStyle)` sets every run in the frame to the style.

### 6.3 `style/layout.py`

Slide geometry for the 13.33" Г— 7.5" widescreen (CSI template):

```python
SLIDE_W, SLIDE_H = Inches(13.33), Inches(7.5)
SAFE_LEFT, SAFE_RIGHT = Inches(0.5), Inches(0.5)
SAFE_TOP, SAFE_BOTTOM = Inches(0.4), Inches(0.4)
TITLE_ZONE   = Zone(top=Inches(0.4), height=Inches(1.0), full_width=True)
FOOTER_ZONE  = Zone(bottom=Inches(0.4), height=Inches(0.4))
CONTENT_ZONE = Zone(top=Inches(1.4), bottom=Inches(0.8))  # between title and footer
KPI_ROW_H    = Inches(1.2)
```

Helpers:
- `fit_image(slide, image_path, zone, preserve_aspect=True)` вЂ” uses PIL, mirrors the pattern already in `deck_builder.py`
- `is_inside_zone(shape, zone) -> bool` вЂ” used to audit whether a title shape sits in the title zone

### 6.4 `style/headline.py`

Scores headlines against `SLIDE_MAPPING.md` rules. **Does not rewrite.** Returns a `HeadlineScore` with:

- `is_complete_sentence: bool`
- `has_metric: bool` (regex for % or $ or number)
- `has_direction: bool` (grew/declined/up/down/rose/fell/etc.)
- `has_driver_clause: bool` (heuristic вЂ” comma or "due to" / "led by" / "driven by")
- `char_count: int`
- `violates: list[str]` (human-readable rule names)

A headline passes if all four booleans are true and `char_count <= 120`.

### 6.5 `style/charts.py`

Chart *images* are already rendered by Python cells (matplotlib PNG at 150 DPI per `SLIDE_MAPPING.md`). Polish cannot re-render. It can only audit. Returns a `ChartAudit` per picture shape:

- `dpi_estimate: int` вЂ” estimated by `image_width_px / rendered_width_inches`; reading image pixel dimensions from the embedded file and rendered width from the picture shape's `width` attribute
- `meets_dpi_floor: bool`
- `bold_color_count: int` (count of non-muted colors in the image; heuristic)
- `violates_focal_rule: bool` (bold_color_count > 1)

Audit-only. No fixes.

### 6.6 `style/narrative.py`

Consultative/performance scorer. For each slide, produces scores 0вЂ“3 on three axes:

- **Consultative** (0вЂ“3): quality of the headline (re-uses `headline.py`)
  - 3 = passes all four headline rules
  - 2 = passes 3 of 4
  - 1 = passes 1 or 2 of 4
  - 0 = no headline or only a fragment
- **Performance** (0вЂ“3): presence of a bottom annotation and a speaker note (no semantic check on "business implication" вЂ” polish only checks that non-empty text exists)
  - 3 = speaker note has non-empty text AND a text shape exists in the `FOOTER_ZONE`
  - 2 = one of the two
  - 1 = speaker note exists but is empty / whitespace only
  - 0 = neither
- **Focal** (0вЂ“3): single visual emphasis
  - 3 = at most one bold color on the slide (across all shapes + images)
  - 2 = two bold colors with clear hierarchy
  - 1 = three bold colors
  - 0 = four or more

Slides scoring <2 on any axis are flagged in the report. Score is informational вЂ” nothing is rewritten.

## 7. Driver вЂ” `polish.py`

### 7.1 CLI

```
polish.py <deck.pptx>                # dry-run (default)
polish.py <deck.pptx> --apply        # write polished deck + report
polish.py --batch <decks_dir>        # run against every .pptx in the folder
polish.py <deck.pptx> --report-only  # skip style fixes, only emit report
```

Flags:
- `--apply` вЂ” write a polished PPTX to `polished/<deckname>.pptx`
- `--out <dir>` вЂ” override output directory
- `--report-only` вЂ” no modifications, only `polish_report.md`
- `--strict` вЂ” exit non-zero if any slide has narrative score <2

### 7.2 What gets force-applied (with `--apply`)

| Fix | Logic |
|---|---|
| Font name | Every run not in Montserrat family в†’ set to the role's font (from `typography.py`). |
| Font size | Only when the shape type maps unambiguously to a role (e.g. title placeholder в†’ slide_title size). Otherwise left alone. |
| Off-palette colors (near-hex match) | Colors within Euclidean distance в‰¤ 5 (in 0вЂ“255 RGB space) of a palette color в†’ snap to the palette color. Colors farther than 5 are treated as intentional and left alone. |

### 7.3 What is flagged only (no auto-fix)

- Headline score <3
- Missing bottom annotation or speaker note
- Multiple bold colors on a slide
- Chart image DPI estimate below 150
- Shapes outside the safe-margin zone
- Missing section dividers (deck-level)
- Non-sequential or missing page numbers (deck-level)
- No client name on title slide (deck-level)

### 7.4 Outputs

Written to `<out>/` (default: `polished/` alongside input):

- `polished/<deckname>.pptx` вЂ” only with `--apply`
- `polish_report.md` вЂ” per-slide findings, deck-level findings, narrative scores
- `polish_diff.md` вЂ” before/after text extraction per slide (text only, not styling)

### 7.5 Report structure

```markdown

- Slides: N

- Flagged (any axis <2): Y
- Force-applied fixes: Z

- **Performance: 3/3**

```

## 8. Testing

### 8.1 Committed fixtures

Three PPTX files live in `02_Presentations/tests/fixtures/`, generated once by a `scripts/make_fixtures.py` helper so they can be reproduced:

- `pristine.pptx` вЂ” 5 slides, all rules pass
- `moderately_broken.pptx` вЂ” 5 slides: one off-palette color, one fragment headline, one missing annotation
- `badly_broken.pptx` вЂ” 5 slides: Times New Roman, red off-palette, no page numbers, no section dividers

### 8.2 Unit tests

- `test_palette.py` вЂ” `nearest_palette`, `is_palette_color`, О” threshold
- `test_typography.py` вЂ” `apply()` mutates every run; roles map to correct sizes
- `test_layout.py` вЂ” `is_inside_zone`, `fit_image` preserves aspect
- `test_headline.py` вЂ” each of the four booleans, edge cases (numbers without %, directionless sentences)
- `test_narrative.py` вЂ” score boundaries (0/1/2/3) per axis
- `test_polish.py` вЂ” integration:
  - `pristine.pptx` в†’ 0 flags, 0 force-applies
  - `moderately_broken.pptx` в†’ known flag count; `--apply` fixes fonts/colors but not headlines
  - `badly_broken.pptx` в†’ similar, verifies force-apply is conservative

### 8.3 No client data

All tests run against committed fixtures. No reference to M: drive or any client-owned PPTX.

## 9. Branch strategy

- `feature/deck-polish` branches from `main`
- Commits are atomic per submodule (`style/palette.py` в†’ test в†’ commit; then `style/typography.py` в†’ test в†’ commit; etc.)
- Driver (`polish.py`) comes after all `style/` submodules pass their tests
- Integration test with fixtures committed with `polish.py`
- PR into `main` when all tests pass and report format is validated against one real deck (user brings a deck locally, runs polish, confirms the report reads right; the deck itself never enters the repo)

## 10. Open questions (none blocking)

3. `SLIDE_MANIFEST.xlsx` is in the repo root. Polish doesn't use it. If it should вЂ” e.g. to validate that a deck contains the slides the manifest expects вЂ” that's a new check. Deferred unless user asks.
